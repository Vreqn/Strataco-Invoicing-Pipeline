Strataco Invoice Automation — Things to Discuss With the Operator
==================================================================

A running list of decisions the code can't make on its own — they
depend on how the business wants invoices handled, named, and filed.
Bug fixes don't end up here; they just get fixed.

Each entry follows the same shape:
  - Question         — one sentence.
  - Example          — a concrete case so you can picture it.
  - Options          — choices and what each one means.
  - Decision         — left blank for Krisztian to fill in.

Newest entries on top. Technical references are at the bottom of each
entry — ignore them unless a developer asks.

2026-05-14  —  AUTO_SPLIT + non-invoice extras: forward to which manager(s)?
--------------------------------------------------------------------------

Question
  When Step 1 routes an email as AUTO_SPLIT (two PDFs for two different
  strata plans in the same email) AND the email also carries non-invoice
  attachments (e.g. a .docx or a plan-less PDF), which manager receives
  the forwarded email?

Example
  An email contains three files:
    - BCS2707 - Invoice.pdf  → auto-split to BCS 2707's manager
    - EPS4280 - Invoice.pdf  → auto-split to EPS 4280's manager
    - Contract.docx          → discarded at intake (not an invoice)
  Both invoices are filed correctly. But who gets the Contract.docx?

Options
  A. Forward to all managers involved in the split (one forward per manager).
     Both managers see the extras; each can decide if it's relevant to them.
  B. Forward to the first split manager only (the first PDF's plan).
     Simpler; one manager is responsible for chasing the other if needed.
  C. Don't forward in AUTO_SPLIT + extras cases at all; flag the email instead.
     Keeps the inbox rule strict; front desk handles it manually.

Decision
  Option A (2026-05-14) — Forward to all managers involved. Implemented as Change D
  in v0.16.0: for every AUTO_SPLIT action, the system collects all unique manager
  emails from per_pdf_plan and sends a separate forward to each.

Technical reference
  _process_self_attachments in steps/step_1_intake.py — currently forwards
  to the first split manager (option B). The AUTO_SPLIT + non-PDF-extras
  combination is rare in practice (AUTO_SPLIT only fires when distinct plan
  bases appear in the same email, which is already unusual).

             only `Processed -` / `Processed-YYYYMMDD-*` files older than
             the 0.17.0 deploy date, runs once, then self-disables. Safer
             than a manual sweep (scope-limited) but more code to write

2026-05-13  —  PDF references an unmanaged strata plan: what happens to it?
--------------------------------------------------------------------------

Question: When a PDF's text explicitly labels a strata plan ("Strata Plan
KAS 9999") that is NOT in Strataplan_List.xlsx, the email is flagged and
held in the Inbox — but there is no defined process for that invoice.
What should happen to it?

Background: 0.15.0 made Step 1 trust the subject's plan when a PDF carries
no plan number at all (the "NO_PLAN" case — see ReleaseNotes.txt). But it
deliberately does NOT do that when the PDF explicitly names a plan that
simply isn't one we manage (the "AMBIGUOUS / Case 3" path). That email
keeps its Outlook red flag and stays in the Inbox. The problem: even if
the front desk replies-to-self with that plan in the subject, the invoice
still can't move — the plan has no manager and no accounts-payable person
in Strataplan_List.xlsx, so Steps 3-6 have nowhere to route or file it.
It would just error further down the line.

Detection note: there is no strict formula for what a strata plan number
looks like, so the system does NOT guess. It only treats a token as a
plan reference when the document literally writes "Strata Plan ..." next
to it. A bare "KAS 9999", or a PO / account / invoice number, is NOT
flagged — it routes on the subject like any ordinary plan-less invoice.
That means the rare unmanaged invoice that names its plan WITHOUT the
words "Strata Plan" will slip through to the subject's manager. Per
Krisztian that is acceptable: these are once-or-twice cases.

Example: An invoice PDF reads "Strata Plan KAS 9999." KAS 9999 isn't on
our plan list — maybe it's a plan another firm manages, a plan that was
closed, or the vendor mistyped it. The subject of the email says a valid
plan we DO manage. Today the email is flagged and sits in the Inbox with
no further path.

  Option A — Front desk asks the vendor to confirm the correct plan.
             If the vendor really meant one of our plans, they resend
             with the right number and it routes normally. If the plan
             genuinely isn't ours, the invoice was misdirected and the
             front desk tells the vendor so. No code change.

  Option B — Add the plan to Strataplan_List.xlsx if it's genuinely one
             we now manage. Once it has a manager + AP, the existing
             pipeline handles it with no special-casing.

  Option C — Define a manual handling path for "real invoice, plan not
             on our list" — e.g. a dedicated folder or a developer
             escalation — for the case where the invoice IS ours to pay
             but the plan list hasn't caught up.

Considered and rejected: an "outdated" folder wired to a placeholder
manager/AP. Krisztian's view — it would catch too much (you can't tell a
genuine outdated plan from noise) and nobody will keep a separate
"outdated plans" sheet current. Also rejected: a hardcoded list of all BC
strata prefixes — same maintenance problem.

How this affects the front desk: under Option A the recipe is "confirm
the plan with the vendor." Options B and C are developer/admin actions.

Interim behaviour (Krisztian, 2026-05-14): keep the email flagged in the
Inbox until the client answers. Most likely outcome is these get
processed manually — they're rare enough that a defined automated path
may not be worth building.

Decision:
  Front desk discretion (2026-05-14). Option A — contact the vendor to confirm
  the plan; if the invoice isn't one we manage, the front desk deletes the email
  or moves it to Handled to break the loop. No code change needed.


  Technical references:
  - steps/step_1_intake.py::_classify_pdf_against_subject — Pass 3.
    NO_PLAN requires BOTH `result.detected` empty AND
    `plan_match.find_explicit_plan_tokens(text)` empty; otherwise
    AMBIGUOUS. This Case 3 is the "explicit token, not in list" branch.
  - tools/_lib/plan_match.py::find_explicit_plan_tokens — the
    managed-list-independent "Strata Plan ..." scanner.
  - steps/step_1_intake.py::_decide_email_action — "any AMBIGUOUS -> FLAG"
    branch.
  - workflows/step_1_intake.md — outcome table + "PDF references an
    unmanaged strata plan" edge case.

2026-05-13  —  Multi-PDF email with suffix-variant plans: route or hold?
--------------------------------------------------------------------------

Background: The system currently flags any email that contains multiple
PDFs whose plans are suffix variants of the same base (e.g. LMS 4193C
and LMS 4193T, or EPS 4280 and EPS 4280A). This is the "plan bases
collide" failsafe — designed to catch vendors who confused two very
similar plans and sent invoices for both under a single subject line.
But when each PDF clearly identifies its own variant (PDF text agrees
with filename per-PDF, and the two PDFs point at different variants),
the flag is conservative and forces the front desk to ask the vendor
to resend the email split per plan.

The distinct-base case (BCS 2707 + BCS 2800 in one email) is already
handled by AUTO_SPLIT — each PDF routes to its own plan with no
intervention. The question is whether suffix variants deserve the
same treatment, or whether the historical risk (vendor labelling
confusion) outweighs the convenience.

Example: Vendor sends one email with two invoices. Invoice A's PDF
text and filename both clearly say "LMS 4193C." Invoice B's both
clearly say "LMS 4193T." Today the system flags the email because
the bases collide ("LMS 4193"). The front desk asks the vendor to
resend as two separate emails; each then routes cleanly. Possible
behaviour: route both automatically, since both PDFs are
unambiguous.

  Option A — Keep flagging (strict). The front desk asks the vendor
             to resend split per plan, or asks for the developer to
             intervene. Safe, but slow whenever a single vendor
             genuinely bills two suffix variants together.

  Option B — Auto-route when each PDF confidently identifies its own
             variant. Each PDF goes to its own manager; subject is
             ignored. Faster, but trusts the vendor's per-PDF
             labelling completely — a vendor who *did* confuse the
             variants would get silently mis-routed.

  Option C — Hybrid. Auto-route only when (1) PDF text agrees with
             filename per-PDF AND (2) the two PDFs point at different
             variants AND (3) no other classifier returned AMBIGUOUS.
             Any uncertainty falls back to flagging. Most conservative
             of the auto-route options.

How this affects the front desk: under Option A (today), the front
desk's recipe for a suffix-variant flag is "ask the vendor to resend
split." Under B or C, the front desk never sees these — the system
handles them.

Decision:
  Option A for now (2026-05-14). Route-per-PDF to respective managers for suffix-
  variant plans with different managers is future work once the overall pipeline is
  stable. Currently, when two PDFs in one email map to suffix variants of the same
  base but point to different managers, the email still flags.


  Technical references:
  - steps/step_1_intake.py::_decide_email_action — "Multiple PDFs,
    plan bases collide" branch is the FLAG path; "Multiple PDFs,
    distinct plan bases" is the AUTO_SPLIT path that B/C would
    model on.
  - plan_match.plan_base reduces "LMS 4193C" and "LMS 4193T" both
    to "LMS 4193" — same convention as xls.base_plan_index.
  - workflows/step_1_intake.md's decision-matrix table documents
    the current behaviour.

2026-05-13  —  Boilerplate-sibling PDF flags: keep strict-first or loosen?
--------------------------------------------------------------------------

Background: the first full live run of Step 1 (2026-05-12 diagnostic
bundles) processed 224 emails and flagged 17 for review. After looking
at every flagged email together with Krisztian, the pattern was clear:
about 8 of the 17 follow this exact shape:

  - Email subject contains a clean plan ID
    (e.g. "FW: BCS3396 Invoice 26376 from Fraser Stratacare Ltd.")
  - Email contains the real invoice PDF (e.g. `Invoice_26376_from_
    Fraser_Stratacare_Ltd.pdf`) — would AGREE with the subject
  - Email also contains a sibling boilerplate PDF (e.g.
    `FSC_Fuel_Surcharge_.pdf`, `Options for Paying Your Invoice NEW.pdf`,
    `FVFP - Customer Payment Options.pdf`, `PPP - Welcome Package.pdf`)
    — boilerplate has strata-ish wording but no clean plan ID
  - Strict-first rule fires (any AMBIGUOUS PDF in an email flags the
    whole email), so the real invoice stays in the Inbox unrouted

The system is working as designed; the question is whether the design
is the right trade-off. Krisztian deliberately deferred this in 0.11.2
and 0.11.3 — the rule was NOT changed.

Example: An email from Fraser Stratacare arrives. The real invoice PDF
clearly belongs to BCS 3396 (subject says so, filename says so, PDF
text says so). Beside it sits `FSC_Fuel_Surcharge_.pdf`, a generic
notice the vendor staples to every invoice. The boilerplate has no
plan ID, the strict-first rule treats it as AMBIGUOUS, and the operator
spends 30 seconds dragging the real invoice into the right manager's
folder by hand.

  Option 1 — Keep strict-first. Today's behaviour. Operator handles ~8
             flag cases per batch caused by boilerplate siblings.
             Pro: zero risk of mis-route. Con: known workload.

  Option 2 — Per-PDF rescue. Change the rule from "any AMBIGUOUS flags
             everything" to "AMBIGUOUS PDFs flag only if no sibling
             AGREEs". An AGREE in the same email rescues the AMBIGUOUS
             sibling, the boilerplate gets parked in `_Unmatched/`, the
             real invoice routes.
             Pro: clears all ~8 boilerplate-sibling cases automatically.
             Con: trusts the AGREE-on-subject signal a bit more than
             strict-first does.

  Option 3 — Known-boilerplate ignore list. Maintain a list of
             filenames known to be informational (`FSC_Fuel_Surcharge_`,
             `Options for Paying Your Invoice NEW`, `FVFP - Customer
             Payment Options`, etc.). These don't participate in the
             cross-check — they're treated like signature images at
             that point. Other AMBIGUOUS PDFs still flag normally.
             Pro: surgical; no change to the safety semantics for any
             other PDF. Con: maintenance — new boilerplates appear
             over time.

  Option 4 — Combination of 2 and 3 — per-PDF rescue plus the ignore
             list — for maximum coverage.

The other ~9 of the 17 are multi-month chase emails where every PDF
inside the email is AMBIGUOUS (the PDF text matcher's safety guard
refuses to pick). Those resolve via the 0.11.2 filename-fallback for
PDFs whose filename has the plan ID, and via the 0.11.3 PDF-text-first
order swap for the rest. So most of the long tail is already gone —
this entry is specifically about the boilerplate-sibling cases.

Decision:
  Resolved in phases. v0.18.0 (2026-05-14) forward protocol handles the primary
  boilerplate-sibling shape: NO_PLAN siblings go to the plan manager via forward,
  not into any strata folder. Front desk inbox stays clean. Sub-question (a)
  (multi-PDF EMPTY siblings) remains open; (b) and (c) are now moot.


--- 2026-05-14 addendum (0.15.0 regression + 0.15.1 partial fix) ---

0.15.0's new PdfOutcome.NO_PLAN inadvertently shipped a behaviour change
against this still-open entry. NO_PLAN was built for "a real vendor
invoice that just never prints its plan number" — but a boilerplate
notice ALSO has no plan number, so NO_PLAN swallowed the boilerplate-
sibling case. On the 2026-05-14 diagnostic run, `FSC_Fuel_Surcharge_.pdf`
(the example above) was classified NO_PLAN, and `_decide_email_action`

Surcharge_.pdf". That is worse than any of Options 1-4 above — none of
them said "stamp the boilerplate and file it as an invoice."

0.15.1 reverts the NO_PLAN half to strict-hold: a NO_PLAN PDF routes on
the subject ONLY when it's the lone attachment; in a multi-PDF email any

shape. Krisztian's stated lean (2026-05-14): multi-attachment emails
where a sibling can't be confirmed should be held — "the key is never

Still open, for Krisztian to settle with his dad / client:

  (b) What is the exact front-desk recovery recipe for a held

  (c) The original Options 1-4 question (per-PDF rescue / known-

      only undoes the accidental loosening, it does not pick an option.

--- 2026-05-14 addendum (0.18.0 — forward protocol replaces strict-hold) ---

0.18.0 changed the NO_PLAN sibling behaviour again. The strict-hold from 0.15.1
is gone: a NO_PLAN PDF in a multi-attachment email is now SKIPPED in routing
(not stamped, not filed), and the full original email (all attachments intact)

This resolves the practical workload problem: managers now receive boilerplate
notices (FSC fuel surcharge sheets, payment-option flyers, etc.) directly in

  (b) Front-desk recovery recipe — no longer relevant for the boilerplate-

      still flags in the Inbox; the recovery recipe there is the developer fixing
      the Strataplan_List.xlsx entry, then letting Step 1 re-process via the next
      run. No reply-to-self needed.

  (c) The original Options 1-4 question (per-PDF rescue / known-boilerplate
      ignore list) is now effectively moot for the boilerplate-sibling shape:
      boilerplate goes to the manager via forward, not into any strata folder.
      The only remaining open case is AMBIGUOUS PDFs in multi-attachment emails —
      those still FLAG as before (AMBIGUOUS is not forwarded).

  (a) Multi-PDF EMPTY siblings — still open. EMPTY in a multi-attachment email
      routes-on-subject (same as pre-0.18.0). If that should instead forward to
      manager like NO_PLAN, that's a one-line code change once decided.

2026-05-11  —  Duplicate-detection: four small policy calls
--------------------------------------------------------------------------

Quick context: 0.9.0 added a duplicate-catcher. It hashes every PDF and
also extracts (plan, invoice number, amount) so it spots the same
invoice arriving 2-3 times in a week — both verbatim resends and
vendor-regenerated copies. Defaults are already in place; the four
questions below are about whether those defaults match how you'd
actually want it to behave.


### Question 1 — Resolved 2026-05-12 (see ReleaseNotes.txt 0.9.1 + 0.9.3).
Bundled the duplicate summary into the morning "Invoices summary"
email (0.9.1), so the question of WHO gets it collapses into "the
same person who gets the daily processed/unmatched report." Added
Step 8 at 18:00 (0.9.3) for the separate end-of-day "stuck in
Inbox" report. Single recipient (notify_email()) for both — no new
env var.


### Question 2 — How should an operator mark a flagged "duplicate" as legitimate?

Today the only way to override a flagged duplicate is the CLI tool
`tools/dup_override.py`, which an operator runs from a terminal with a
typed `--reason`. That assumes the operator is comfortable opening a
command-line window — which the people actually processing invoices
won't be. They live in Outlook and follow a short training brief.

The CLI stays useful as a developer escape hatch (Krisztian can still
run it directly), but we need an operator-facing gesture in Outlook
itself so the real users can actually do this.

Example: A vendor sends a bill that gets flagged as a duplicate of
last month's invoice — but it's actually a fresh re-bill with a
corrected amount. The flagged email lands in `Inbox/duplicate_emails`
and shows up in tomorrow's summary. The operator needs a way to say
"no, route this one normally" without leaving Outlook.

  Option 1 — Reply to the daily duplicate-summary email.
             The summary already lists each duplicate with a numbered
             handle and full context (plan, invoice #, amount, sha256
             prefix). The operator replies with one line per override:
                 Allow #2 — BC Hydro reissued with corrected amount.
                 Allow #5 — new bill, vendor reused an old invoice #.
             The system parses the reply on the next intake, flips the
             matching ledger rows to `overridden`, and stores the
             reply body as the audit reason.
             Pro: batch-friendly; the audit reason is captured for
             free because the reply IS the reason.
             Con: operator has to wait for the next morning's summary
             before they can override.

  Option 2 — Reply to the flagged duplicate email itself.
             The flagged email lives in `Inbox/duplicate_emails`. The
             operator opens it, hits reply-to-self (same muscle memory
             as the existing routing trick), and puts `OVERRIDE` in
             the subject: `OVERRIDE: BCS 2707 — Re: …`. Reply body
             becomes the audit reason.
             Pro: closest fit to the reply-to-self pattern operators
             already know; can be done the moment a duplicate is
             noticed, not next-morning.
             Con: one reply per override; no batching.

  Option 3 — Drag the email to a "release" folder.
             Create a new Outlook folder `Inbox/duplicate_release`.
             The operator drags the flagged email into it. Next intake
             run, the system re-routes its PDFs and marks the
             fingerprints overridden.
             Pro: simplest possible gesture, zero typing.
             Con: no audit reason captured — the log just says
             "operator released this on date X" with no "why."

  Option 4 — Tiered: drag for routine, reply for high-stakes.
             Drag-to-folder works for duplicates flagged at the
             `manager_queue` or `ap_queue` stage (routine catches
             before the invoice was paid). For overriding a row in
             the `archived` stage (re-doing a closed transaction) the
             system refuses the drag and requires a reply-with-reason
             instead. Matches the spirit of the old Q2's Option 3 —
             extra friction only where it matters.
             Pro: low friction for routine cases, strong audit for
             risky cases.
             Con: more moving parts; two gestures to train.

  Sub-question (applies inside whichever option you pick): should an
  audit reason be REQUIRED, OPTIONAL, or SKIPPED? Reply-based options
  (1, 2, 4-archived) capture a reason naturally. Drag-based options
  (3, 4-routine) capture only "who and when." If you care about audit
  for every override, that rules out Option 3 and the routine half of
  Option 4.

Decision:
  Deferred (2026-05-14). The 4-field fingerprint (v0.9.2) already handles the
  common re-bill case: a corrected amount changes the fingerprint so the re-bill
  is not flagged as a duplicate. An Outlook-based front-desk gesture for the rare
  edge case (same plan + invoice# + amount + domain, genuinely different invoice)
  is open for a future sprint.


### Question 3 — Resolved 2026-05-12 (see ReleaseNotes.txt 0.9.2).
Added sender_domain as a 4th deterministic Layer B factor — no LLM,

### Question 4 — Resolved 2026-05-12 (see ReleaseNotes.txt 2026-05-12 doc-only entry).

2026-05-11  —  Step 7 (Monthly Aggregator): what to do with "late checks"
--------------------------------------------------------------------------

### Question — When a check arrives AFTER the month's summary has already been sent, should the system produce a second summary, replace the original, or refuse?

Step 7 rolls up each strata plan's archived invoices for the month
into one "Summary" PDF. Trouble is: an invoice paid right at the end
of the month sometimes lands AFTER Step 7 has already run.

Example: Step 7 ran on May 7th and emailed the April Summary to the
board. On May 8th, an April-30th invoice finally gets archived.
Today the system writes a SECOND file called
"Summary - 04 - <plan> April 2026 inv (1).pdf" containing just that
late check — so the board may receive two Summary emails for the
same month.

  Option 1 — Additive `(1)` Summary (current default). Safe and
             reversible. Operator decides whether to delete the
             original, delete the (1) variant, or send both. Ledger
             row: "aggregated_late".

  Option 2 — Destructive re-merge. Step 7 reads the existing Summary,
             prepends/appends the late check, writes a single
             replacement Summary, deletes the original. Cleanest end
             state. Risky if the original was already shared
             externally — the link goes stale.

  Option 3 — Refuse re-merge once a Summary exists. The operator must
             explicitly re-run with `--force` (and possibly delete the
             old Summary by hand) to bring in late checks. Safest in
             terms of "no surprises," but moves the late-check problem
             to the operator's monthly checklist — and the late check
             keeps piling up in the plan folder until the operator
             notices.

Decision:
  Option 1 (2026-05-14) — Keep the additive "(1)" file. Safe and reversible;
  front desk decides whether to send both summaries to the board or discard one.
  Anything that arrives after the month's summary was sent is processed for that
  respective month regardless — no re-merging needed.

2026-05-11  —  Plan numbers with letter suffixes (A, B, …)
--------------------------------------------------------------------------

Quick context: some properties have suffixed plans like BCS 2707A and
BCS 2707B (two separate buildings, sometimes same manager). The bare
"BCS 2707" can also be its own plan. PDFs sometimes use the suffix,
sometimes don't. Two related questions below.


### Question 1 — If a PDF clearly says "BCS 2707A", should that ALSO be treated as a mention of plain "BCS 2707"?

Right now it counts as both. So if your plan list has BOTH "BCS 2707"
and "BCS 2707A" as active plans, the system gives them equal credit
and flags the invoice as ambiguous — even though a human would
obviously pick 2707A.

Example: A vendor PDF reads "Property: BCS 2707A — Building B."
Today the system records one mention each for 2707A and 2707, ties,
and dumps the file into the Ambiguous bucket.

  Option 1 — Strict: a mention of "BCS 2707A" only counts for 2707A.
             Cleanest. Some files that currently land in Ambiguous
             would start routing correctly. Small risk: edge cases
             where the suffix in the PDF was a typo would now route
             wrong instead of asking.

  Option 2 — Weighted: still count it as a mention of the base, but
             give the exact suffixed match a higher score so it
             always wins. Same end result as Option 1 in practice,
             just achieved differently. Slightly harder to reason
             about later.

  Option 3 — Leave it alone. Accept that occasional ambiguous flags
             will happen when a base and its variant both exist in
             the plan list. Document the trap so operators know to
             check those manually.

Decision:
  Option 1 (2026-05-14) — Strict. "BCS 2707A" only counts for "BCS 2707A."
  Implemented in v0.16.0: the double-scoring of the base plan was removed from
  match_from_pdf_text, eliminating this class of avoidable flags.


### Question 2 — If a PDF says just "BCS 2707" but only "BCS 2707A" and "BCS 2707B" exist on the plan list, what should the FILENAME say?

Routing is fine in this case — if 2707A and 2707B share a manager, it
goes to that manager. The question is what to write on the file.
Today it picks whichever variant comes first in the list and labels
it that way.

Example: Vendor PDF clearly says "BCS 2707." Plan list has 2707A and
2707B under Sarah. The file gets saved as
"BCS 2707A - ABC Plumbing.pdf" — even though the invoice itself only
ever said 2707.

  Option 1 — Route by manager, label by what the PDF actually said.
             File gets saved as "BCS 2707 - <vendor>.pdf" even though
             it lives in the manager's 2707A/B workflow. Honest to the
             source document.

  Option 2 — Keep current behaviour: route AND label by the first
             variant that shares the manager. Convenient but the
             filename misrepresents what was on the invoice.

  Option 3 — Refuse to guess. Drop the file into the Ambiguous bucket
             and make a human pick. Safest, but adds operator work
             every time.

Decision:
  Option 3 (2026-05-14) — Flag for front-desk correction. When the PDF names only
  a base plan and the list has only variants, the system flags and holds so the
  front desk can confirm the correct variant. Implemented as AMBIGUOUS via
  is_base_fallback in v0.16.0.


### Real-world cases to test against once you've decided

Whatever you pick, we should walk through these four scenarios with
your answer in hand and confirm the system does the right thing before
shipping the change:

  1. PDF says "BCS 2707A", plan list has "BCS 2707A" — should route
     to that exact plan's manager. (Sanity check.)

  2. PDF says "BCS 2707", plan list has "BCS 2707A" and "BCS 2707B"
     under the same manager — where should it go, and what should the
     filename be?

  3. PDF says "BCS 2707", plan list has BOTH "BCS 2707" (no suffix)
     AND "BCS 2707A" as separate active rows — which one wins?

  4. PDF says "BCS 2707A", plan list has "BCS 2707A" and "BCS 2707B"
     under DIFFERENT managers — should stay unmatched. (Already works
     correctly today; mentioned so we don't regress it.)

Strataco Invoice Automation — Release Notes
============================================

Newest entries on top. Each version corresponds to the value in `VERSION`.
Bump the version on every meaningful change before committing.

Versioning: MAJOR.MINOR.PATCH
  - PATCH  — bug fixes, log/text tweaks, no behaviour change
  - MINOR  — new feature, new step, new tunable knob, backwards-compatible
  - MAJOR  — breaking change to file paths, .env keys, or step contracts

0.12.1  —  2026-05-13  —  Codex review follow-ups on the 0.12.0 stamp flatten
---------------------------------------------------------------------------

Codex adversarial review of 0.12.0 surfaced seven findings. This release
addresses six of them; the seventh (viewer-specific saves with /AP but no
/V) is deferred because Acrobat and Acrobat Reader — the operator's
tools — both populate /V on save.

Fixes:

  A. **Vendor /AcroForm now survives intake.** 0.12.0's
     `_merge_overlay_onto_page_one` flattened the base PDF's AcroForm
     before merging any new stamp overlay. That was a latent regression:
     a vendor sending a fillable invoice would have their form stripped
     on intake (Step 1 / Step 3), making it unfillable for the manager
     or AP. The flatten call has been removed from the merge path
     entirely. Step 1 and Step 3 now leave vendor forms intact.

     The orphan-widget bug the intake flatten was masking is fixed
     directly: new helper `_register_existing_widgets_in_acroform()`
     promotes any widget annotations on the merged page into the
     writer's freshly-created `/AcroForm/Fields`, so vendor and prior-
     stage widgets stay registered (and fillable in all viewers).

  B. **Step 5 now flattens explicitly before the Paid stamp.** Moved
     out of the merge helper and into `steps/step_5_to_ap.py` directly.
     The manager-filled Received-stamp values (and any vendor /AcroForm
     fields the manager touched) get baked as static text BEFORE the
     editable Paid stamp is added. By the time the AP sees the file,
     the manager's data is locked.

     If `flatten_acroform` raises, the source stays in the manager's
     `Approved/` for the next run rather than being saved unflattened.

  C. **Step 6 fails closed on flatten error (Codex #3).** 0.12.0's
     `_archive_one` fell back to writing the unflattened bytes if
     `flatten_acroform` raised, AND still deleted the AP source —
     producing a permanent archive with editable widgets and no
     recovery path. Now an exception surfaces as a `paid_failed` entry
     in the morning email, the archive is not written, and the AP
     source stays in place for the next run.

  D. **Ledger gains `archive_sha256` (Codex #4).** Step 6's flatten
     changes the on-disk byte hash, so `tools/dup_reconcile.py` —
     which hashes archived files and looks them up by SHA — used to
     report every flattened archive as an orphan. Added an
     `archive_sha256` column to the duplicate ledger. Step 6 computes
     the post-flatten hash and stores it alongside the chain SHA.
     `dup_reconcile` now matches against either column. Pre-0.12.1
     CSVs parse cleanly via the same migration pattern as 0.11.2's
     `need_review` column.

  E. **`extract_paid_stamp_values` corrections (Codex #5, #6, #7):**

       * Authoritative AcroForm tier: when `/V` returns any non-empty
         value, the function short-circuits. Partial fills (one paid_*
         set, the other empty) no longer fall through to positioned
         text — that fallback could otherwise grab a vendor "Date:"
         or "Check Number:" by mistake. The caller honestly reports
         the missing field and the operator fills it.
       * Image-only detection now scopes to page 1 only. A rasterised
         page 1 + text-bearing page 2 (the actual shape of the VR 115
         test invoice) is correctly flagged as `image_only`.
       * Regex fallback also scopes to page 1 via the new
         `extract_page_text(pdf_bytes, 0)` helper in
         `tools/_lib/pdf_text.py`. A remittance stub or terms page
         later in the document with "PAID — Check Number: …" no
         longer leaks into the regex match.

Tests:

  `tests/test_stamp_read.py` — eight new cases, all passing:

    1. `vendor_acroform_survives_intake` — Strataco stamp widgets and
       vendor `vendor_email` widget both editable after intake; no
       flatten.
    2. `step5_flatten_locks_vendor_and_received` — manager-filled
       vendor field AND Strataco fields baked as text after the
       explicit Step 5 flatten; only Paid widgets editable.
    3. `step6_fails_closed_on_flatten_error` — monkeypatch
       `flatten_acroform` to raise, run `_archive_one`, assert the
       file is flagged paid_failed, source not deleted, no archive
       written.
    4. `archive_sha256_round_trip` — chain SHA and archive SHA differ,
       both persist through the ledger.
    5. `dup_reconcile_via_archive_sha` — orphan check uses both SHA
       indexes.
    6. `multipage_image_only` — 2-page PDF (rasterised page 1 + text
       page 2) flags as `image_only`.
    7. `partial_acroform_fail_closed` — `paid_check_number_*` filled,
       `paid_date_*` empty: returns the check number, empty date,
       NO positional fallback.
    8. `multipage_regex_no_false_positive` — page-2 "PAID — Check
       Number: 99999999" does not leak into the regex fallback.

  Existing 0.12.0 test (`acroform_round_trip`) updated to model the
  new Step 5 flow (explicit `flatten_acroform` before `render_paid_stamp`).

Deferred:

  * **Codex #2** — viewer-specific saves where the typed value lives
    in widget `/AP` but `/V` is missing/stale. Acrobat and Acrobat
    Reader both set `/V` on save; the realistic exposure is a
    third-party / mobile / online viewer that the operator isn't
    expected to use. Re-evaluate if a real case appears in the field.

Recovery for any in-flight invoice damaged by 0.12.0:

  Any invoice that went through Step 5 between the 0.12.0 ship and the
  0.12.1 ship had its Received-stamp values stripped by the now-removed
  intake flatten. The pre-Step-5 manager-filled bytes are preserved as
  `Users/<Manager>/Invoices/Approved/Processed - <name>.pdf`. To
  recover one: drop the `Processed - ` prefix to restore the original
  filename, re-run Step 5. The 0.12.1 flow flattens correctly.

0.12.0  —  2026-05-13  —  Stamp flatten via pikepdf; AcroForm-direct read in Step 6
---------------------------------------------------------------------------

End-to-end test of the pipeline turned up two layered bugs around the
Received / Paid stamps, both fixed here. The operator-facing change is the
simplest part: there is no longer a "Print → Save as PDF to flatten" step.
The AP fills the Paid stamp in Acrobat, hits Ctrl+S, drags the file into
Paid_Invoices/. Step 6 reads the values straight from the AcroForm and
flattens automatically on archive.

Background:

  * Bug 1 — Step 6 couldn't read Microsoft Print-to-PDF output. On a
    Windows machine, "Print → Save as PDF" rasterises any form-bearing
    page into JPEG strips. The resulting PDF has zero extractable text
    and no /AcroForm, so the positioned-text extractor returned empty
    and the file landed in `paid_failed` with the generic "Could not
    read Check Number" message.

  * Bug 2 — Step 5 was silently dropping the manager's Received-stamp
    values. `tools/_lib/stamp.py::_merge_overlay_onto_page_one` used
    `PdfWriter().append_pages_from_reader(base)` which copies pages but
    not the document-level /AcroForm. `_ensure_acroform` then created a
    fresh empty AcroForm and `_migrate_widget_annotations` only added
    the new overlay's widgets. The base PDF's Received-stamp widgets
    orphaned out of /AcroForm/Fields. Reproduced in isolation: a
    manager-filled PDF with 5 Received-stamp fields became a 2-field
    Paid-only PDF after `render_paid_stamp`.

  * Workflow problem — both `HANDOFF.md` and
    `workflows/step_6_paid_archive.md` told operators to "Print → Save
    as PDF" to flatten. On Windows that is Microsoft Print to PDF (= Bug
    1). The operator was correctly following written instructions.

Fixes:

  A. **New `pikepdf` dependency for reliable text-preserving flattening.**
     `requirements.txt` adds `pikepdf` (wraps QPDF). pypdf is unchanged
     for everything else. `tools/_lib/stamp.py::flatten_acroform(bytes)`
     calls `generate_appearance_streams()` + `flatten_annotations(mode='all')`
     to bake each widget's /V into the page content stream and strip the
     widget annotations. Returns the original bytes unchanged when the
     PDF has no AcroForm (so Step 1 / Step 3 paths on fresh vendor PDFs
     are unaffected).

  B. **Step 5 flattens the Received stamp before adding the Paid stamp.**
     `_merge_overlay_onto_page_one` now calls `flatten_acroform` on the
     base PDF first. The manager's Received-stamp values become static
     text on the page, the widget-orphan bug stops mattering (the base
     has no widgets to lose), and the new Paid-stamp overlay adds the
     only AcroForm fields in the result. AP sees a PDF where Received
     values are visible-but-not-editable and only the Paid Date + Check
     Number fields are editable.

  C. **Step 6 reads AcroForm /V directly, then flattens the archive.**
     `tools/_lib/stamp_read.py::extract_paid_stamp_values` is now tiered:
       1. AcroForm read — pypdf `get_fields()` for `paid_check_number_*`
          and `paid_date_*`. Normal happy path for the 0.12.0 flow.
       2. Positioned text in the PAID region — unchanged, covers PDFs
          that have been flattened by a text-preserving tool (Acrobat
          Pro Flatten Form Fields, Kofax, prior pipeline runs).
       3. Regex over the full text — unchanged.
     `steps/step_6_paid_archive.py::_archive_one` calls `flatten_acroform`
     on the AP-folder bytes before `safe_write_unique` to the archive.
     The archived copy in `Strata_Plans/<plan>/` has zero editable
     fields. The `Processed - <name>.pdf` marker in the AP folder keeps
     the original (unflattened) bytes so the operator can re-open and
     verify in Acrobat if needed.

  D. **Clearer error for image-only PDFs.** `PaidStampValues` gains an
     `image_only` flag, set when AcroForm is absent AND `extract_full_text`
     returns "". Step 6 surfaces this as a dedicated paid_failed reason:
     "PDF appears image-only (likely Microsoft 'Print to PDF') — re-save
     by just hitting Ctrl+S, do not flatten manually". Previously these
     PDFs got the generic "Could not read Check Number" message.

  E. **Workflow docs updated.** `workflows/step_6_paid_archive.md` and
     `HANDOFF.md:57` drop the "Print → Save as PDF" instruction; the new
     wording is "fill in the form fields, save with Ctrl+S, drag to
     Paid_Invoices — Step 6 flattens automatically on archive."

Tests:

  `tests/test_stamp_read.py` — two new cases:

    1. `_check_acroform_round_trip` — end-to-end with no shortcut:
         * `render_received_stamp` on a synthetic invoice
         * fill Received fields via pikepdf (simulates Acrobat Save-As)
         * `render_paid_stamp` (which now flattens Received first)
         * assert Received values are static text, only 2 paid_* fields
           remain as editable widgets
         * fill Paid fields via pikepdf (simulates AP Save-As)
         * `extract_paid_stamp_values` returns the AP-typed values via
           the new AcroForm tier; `image_only` is False
         * `flatten_acroform` on the AP-filled PDF → zero editable fields
           in the archive; every value (Received + Paid) is extractable
           as static text

    2. `_check_image_only_detection` — synthesises an image-only PDF
       (JPEG XObject + content stream, no /AcroForm, no text) and asserts
       `extract_paid_stamp_values(...).image_only` is True.

  The existing reportlab-flatten cases still pass — they exercise the
  positioned-text fallback path.

Recovery for any invoice that Step 5 already stripped pre-0.12.0:

  Each `Users/<Manager>/Invoices/Approved/Processed - <name>.pdf` holds
  the pre-merge manager-filled bytes (saved by Step 5 BEFORE the
  destructive merge — these are not flat, they still have the Received
  stamp's /V values intact). To recover one:
    1. Delete the stripped copy from the AP's Approved_Invoices/.
    2. Rename the `Processed - <name>.pdf` to drop the prefix.
    3. Re-run Step 5 — Step 5 now flattens the Received fields properly
       and the AP gets a correctly-stamped PDF.

  Done as a manual operator step; no scripted backfill.

0.11.3  —  2026-05-13  —  Codex review follow-ups on the 0.11.2 Step 1 work
---------------------------------------------------------------------------
Codex adversarial review of 0.11.2 returned verdict DO-NOT-SHIP with one
blocker and four risks. Full review filed at
`codex-router-out/2026-05-13-step1-0.11.2-review/codex-review.md`. User
triaged each finding; this release addresses findings 1, 3, 4, and 5.
Findings 2 (real PDF discarded on weak Graph metadata) and 6
(`_process_prior_attachments` PDF-only filter) were explicitly accepted
as-is for now.

Fixes:

  A. **Magic-byte PDF validation (blocker, finding 1).** Adds
     `_is_real_pdf(blob)` in `steps/step_1_intake.py` — bytes must start
     with `%PDF-` to be accepted. Wired in at every download site:
       * `_process_self_attachments` Pass 2 — a new `invalid_pdfs` list
         catches `.pdf`-named imposters (PNG/JPEG/HTML/Word bytes) and
         flags the email via `run.review()` instead of letting an EMPTY
         classification route the bytes as the subject.
       * `_process_pdf_text_fallback` Pass 1 — same `invalid_pdfs` list
         gates `can_route_all` alongside `download_failures`; the
         all-or-nothing detail string surfaces the invalid filenames.
       * `_process_prior_attachments` — appends the invalid name to
         `download_failures` with an `(invalid PDF bytes)` suffix so the
         existing conversation-link flag path triggers.
     Scanner-made image-only PDFs still pass (the PDF wrapper carries the
     header regardless of what's inside).

  B. **PDF-text-first, filename as fallback (risk, finding 3).** Per user
     direction, the evidence priority in `_classify_pdf_against_subject`
     and the `_process_pdf_text_fallback` per-PDF loop is swapped: PDF
     body text is the primary signal, filename is consulted only when
     text doesn't yield a confident match. This catches the case where a
     vendor renames a BCS 3396 invoice to "EPS 6008 invoice.pdf" to
     match the wrong subject — pre-0.11.3 it would have AGREE'd via
     filename and routed to EPS 6008; post-0.11.3 the PDF text wins,
     outcome is CLASH, email flags for review. Existing 0.11.2 tests
     still pass (the filename fallback fires on empty/ambiguous text).
     New test `test_pdf_text_wins_over_conflicting_filename` covers the
     Codex repro directly.

  C. **`daily_summary.csv` migration concurrency + Excel-lock safety
     (risk, finding 4).** `tools/_lib/log.py`:
       * `summary.exists()` now checked INSIDE the portalocker lock so
         two concurrent first-time runs on a fresh log dir can't both
         write a header.
       * `_migrate_summary_if_legacy` returns `bool` and wraps both the
         read and the `tmp.replace()` in try/except OSError. If
         migration fails (Excel has the file open), the function
         returns False, the caller logs a WARNING and skips the append
         rather than corrupting the file by appending a 7-column row
         to a still-6-column file.
       * `_append_summary_row` itself is wrapped in try/except OSError
         so a CSV write failure logs a WARNING and returns rather than
         crashing the step from its `finally` block.
     The temp file (`.csv.migrating`) is cleaned up on replace failure.

  D. **Partial-commit Outlook flag guard in the PDF-text fallback path
     (risk, finding 5).** `_process_pdf_text_fallback` now mirrors the
     guard already present in `_process_self_attachments`: when any
     attachment in an email fails after others succeeded, the email is
     flagged with the Outlook to-do marker before being left in the
     Inbox. Without this, the email looked indistinguishable from
     un-processed mail despite having committed PDFs downstream.
     Triggers when, e.g., the PDF routes successfully but a sibling ZIP
     download or write fails.

Tests:

  `tests/test_step1_filename_first.py` grew from 5 to 7 scenarios:
    + `test_pdf_text_wins_over_conflicting_filename` (finding 3 repro)
    + `test_is_real_pdf_magic_byte_check` (finding 1 helper)
  Existing 5 tests updated for the new order (their outcomes were
  identical pre- and post-swap because empty/ambiguous text falls back
  to filename either way).

Out of scope (accepted as-is):

  - Finding 2 — a real PDF with weak Graph metadata (name="12345",
    contentType="application/octet-stream", no invoice hint) is
    discarded at categorisation. Niche enough to defer.
  - Finding 6 — `_process_prior_attachments` still uses metadata-only
    filtering for "is this a PDF?". The 0.11.3 magic-byte check applies
    inside this function for bytes that ARE downloaded, but the
    pre-download filter retains its existing semantics.

0.11.2  —  2026-05-12  —  Step 1 intake: TELUS ZIP fix, filename-first matching,
                          PDF+ZIP-only filter, need_review reporting
---------------------------------------------------------------------------
First end-to-end run of Step 1 against the live Inbox (2026-05-12 diagnostic
bundles) surfaced four bugs / reporting gaps plus one documentation issue.
All five are addressed here. The strict-first ambiguous-PDF rule at
`_decide_email_action` is INTENTIONALLY unchanged — it remains an open
client risk-tolerance conversation.

Fixes:

  A. **TELUS ZIP attachments no longer fall through to "non-PDF/ZIP".**
     `steps/step_1_intake.py` Pass 1 categorisation gated on `ext == ".pdf"`
     before the `_looks_like_pdf_or_zip()` check, so ZIPs were sorted into
     the non-PDF bucket and either triggered the all-or-nothing hold
     (`_process_pdf_text_fallback`) or were saved as plain files instead
     of being staged for Step 2 to unpack (`_process_self_attachments`).

     Both functions now keep a dedicated `zip_atts` bucket. On the happy
     path, ZIPs are downloaded and saved to `_Unmatched/Invoices/` (where
     Step 2 picks them up). In `_process_pdf_text_fallback`, ZIPs no longer
     gate the all-or-nothing decision — only unmatched PDFs and download
     failures do.

  B. **Signature images and one-off office files are discarded at intake.**
     Outlook signature decorations (`image001.png`, `Outlook-*.png`,
     `ServiceBox-Navy.png`, `~WRD*.jpg`) and `.docx` / `.xlsx` attachments
     are now logged at INFO and dropped — not downloaded, not written to
     `_Unmatched/`. This unsticks the ~9 emails per run that previously
     hit all-or-nothing solely because of a sibling decoration image.

     Per client direction, `.docx` / `.xlsx` invoices are accepted as part
     of this trade-off (vendors are being asked to send PDFs going forward).

  C. **Filename-first plan matching in Step 1.**
     `_classify_pdf_against_subject` and the per-PDF loop in
     `_process_pdf_text_fallback` now try
     `plan_match.match_from_filename_with_base_fallback()` BEFORE
     `extract_full_text()` + `match_from_pdf_text()`. Vendor-authored
     filenames (e.g. `EPS 6008 - April 2026 Invoice #17942.pdf`,
     `Kimberly Court LMS 1624 - April 2026 Invoice #17852.pdf`) are
     treated as authoritative when they yield an active-plan match —
     same helper Step 3 already uses, so the two stages agree.

     Expected effect on the 17 strict-flagged emails: the multi-month
     "consensus AMBIGUOUS" cases (EPS 6008, Kimberly Court LMS 1624,
     LMS 3011 Aspen Gabels) become all-AGREE and route. The "boilerplate
     sibling" cases (FSC_Fuel_Surcharge_, Options for Paying Your Invoice
     NEW, FVFP - Customer Payment Options) still flag — those filenames
     have no plan ID and the strict-first rule is unchanged.

  D. **`need_review` counter replaces the misleading `errors=17` headline.**
     `tools/_lib/log.py` `_Run` now carries a `need_review` list alongside
     `processed` and `errors`. New `run.review(msg)` method appends and
     logs at WARNING with a `NEED_REVIEW:` prefix. The three "left in
     Inbox for human review" code paths in Step 1 now call `run.review()`
     instead of `run.error()` / `run.info()`:
       1. `FLAG_AND_HOLD` (strict-first ambiguous / clash) — was `run.error`.
       2. `all-or-nothing` (PDF-text fallback can't route everything) — was `run.info`.
       3. Subject-matched-but-no-eligible-prior-conversation — was `run.info`.

     The `skip` paths (no plan + no attachments) stay on `run.info()` —
     not counted under need_review per client direction. True exceptions
     (download failures, Graph API failures, ledger errors) stay on
     `run.error()`.

     Run summary line changed from:
       `processed=N errors=M duration=Xs status=...`
     to:
       `processed=N need_review=R errors=M duration=Xs status=...`
     `daily_summary.csv` gains a `need_review` column between `processed`
     and `errors`. A one-shot migration in `_append_summary_row` pads
     pre-0.11.2 rows with `0` so historical data is preserved.

  E. **Step 2 / Step 3 wording in the client training brief.**
     `docs/client_training_brief.md` previously said Step 2 "Unpacks any
     ZIP attachments" without specifying *where* it reads from. Operators
     reasonably read that as "ZIPs in the Outlook Inbox" — actual code
     reads `_Unmatched/Invoices/` only. Rewritten to explicitly call out
     the folder, note that Step 2 does NOT touch Outlook, and clarify
     that Step 3 sorts ANY PDF in `_Unmatched/Invoices/` (not just
     zip-extracted ones).

Tests:

  `tests/test_step1_filename_first.py` (new) — five scenarios:
    1. filename match returns AGREE without invoking text extraction
    2. filename plan != subject -> CLASH
    3. filename plan with no manager -> falls back to PDF text
    4. boilerplate filename (no plan ID) -> falls back to PDF text
    5. `run.review()` separation from `run.error()`

Out of scope (deferred):

  - The strict-first AMBIGUOUS rule in `_decide_email_action` is unchanged.
    Boilerplate sibling PDFs continue to flag the whole email. Client risk-
    tolerance conversation TBD.

0.11.1  —  2026-05-12  —  Codex review fixes for the 0.11.0 consolidation
---------------------------------------------------------------------------
Adversarial review by Codex on the 0.11.0 work surfaced one blocker and
three risks. All four are addressed here. Full review filed at
`codex-router-out/2026-05-12-step8-consolidation-review/codex-review.md`.

Fixes:

  A. **Filesystem scans no longer kill the morning email contract.**
     The 0.11.0 release wrapped the Inbox query in try/except but the
     two new pipeline-residue scans (`_scan_unmatched_intake`,
     `_scan_manager_stuck`) could still raise on a permission error and
     take down the whole `main()`, producing no email at all.

     Each scan now returns a new `_ScanResult` dataclass (`rows` +
     `errors`). Errors are collected per-folder (so one bad manager
     doesn't hide the rest of the managers), passed through to the
     email builder, and surfaced in a new Action Required sub-section
     `-- Pipeline scan errors (investigate folder permissions) --`.
     `main()` also wraps each scan call in a belt-and-braces try/except
     so a future non-defensive code path still produces an email.

     The "always-sent" contract Krisztian explicitly approved in 0.11.0
     is now load-bearing again.

  B. **Inbox-query failure now correctly counts in the subject.**
     Before: `Subject: ... 0 action required, ...` even when the body
     said "Inbox query failed — open Outlook directly." An operator
     scanning subjects could miss the degraded Inbox status.

     After: `inbox_error` adds +1 to `action_count`, so the subject
     reflects exactly one item needing investigation. Same fix applies
     to each entry in `scan_errors`.

  C. **Manager-stuck scan is now glob-based (XLS-drift-safe).**
     Before: `_scan_manager_stuck(rows)` iterated
     `unique_managers(rows)` from the Strataplan snapshot. If the
     snapshot was empty/malformed or all rows inactive (rare but real
     during XLS edits), the scan returned `[]` even when manager
     `Approved/` folders had stuck PDFs.

     After: `_scan_manager_stuck()` (no args) globs
     `<root>/Users/*/Invoices/Approved/*.pdf` directly. Manager name
     comes from the directory segment; the scan is independent of the
     XLS state. AP-only `Users/<X>/Paid_Invoices/` entries are
     silently skipped (no `Invoices/Approved/` subdir).

     The `unique_managers` import was dropped from
     `steps/step_6_paid_archive.py` since this was its only caller.

  D. **Test coverage for the scan helpers and degraded paths.**
     Codex pointed out that the 0.11.0 test suite would have passed
     even with the BLOCKER above present. Added:

       - `tests/test_step6_scans.py` (new, 13 cases, 38 assertions):
         covers missing folder, empty folder, mixed contents, unicode
         filenames, iterdir/glob error trapping (per-folder isolation),
         AP-only Users entries ignored, and the disk-only-no-XLS-row
         case that proves the glob-based fix from (C).
       - `tests/test_step6_summary_email.py` (3 new cases, 16 new
         assertions): `inbox_error` counts as 1 in the subject;
         scan_errors render their own sub-section AND count; clean day
         renders exactly one `None today.` under Action Required.

     The 0.11.0 test asserting `0 action required` when `inbox_error`
     is set was locking in the bug — updated to the correct
     `1 action required` after the (B) fix.

Files:
  - Modified: `steps/step_6_paid_archive.py` (new `_ScanResult` dataclass,
    rewritten scan helpers, updated `main()` and
    `_build_combined_summary_email`).
  - Modified: `tests/test_step6_summary_email.py` (new signature,
    corrected inbox_error subject, 3 new cases).
  - Added: `tests/test_step6_scans.py`.
  - Modified: `workflows/step_6_paid_archive.md`
    (Outputs section documents the new Pipeline scan errors sub-section).
  - Modified: `VERSION` → 0.11.1, `HANDOFF.md` (version line).
  - Modified: `ReleaseNotes.txt` (this entry).

No new `.env` keys. No operator-visible behavior change in normal (clean)
operation. Degraded modes (Graph failure, folder permission error) now
produce a more accurate subject line and a dedicated body sub-section.

0.11.0  —  2026-05-12  —  Consolidate Step 8 into Step 6's morning report
---------------------------------------------------------------------------
Operator now receives ONE automation email per weekday, not two. Step 6 at
07:00 is the single morning surface for everything that needs attention.

0.10.1  —  2026-05-12  —  Step 1 commit-path hardening (Codex review fixes)
--------------------------------------------------------------------------
Codex reviewed 0.10.0 and found seven issues, all on the **commit path**
(what happens to message state after `_decide_email_action` returns).
The decision matrix itself was correct; the bugs were that the matrix's
verdict didn't always translate to operator-visible state. Each fix
preserves strict-first: when something goes sideways, flag the email
loudly rather than let it look like ordinary mail.

Fixes:

  A. **Self-attachment partial-batch failure now flags.**
     `_process_self_attachments` — if any attachment in the batch
     returned `FAILED` after others routed, the email now gets the
     Outlook to-do flag with a `partial commit on '<subject>': N of M
     attachments committed before F failure(s)` error log. Committed
     files are already on disk + in the duplicate ledger (no rollback)
     — the flag + log are the operator's reconciliation signal.
     Previously: email sat in the Inbox unflagged, indistinguishable
     from ordinary mail; next runs duplicate-skipped the success and
     re-failed the failure forever.

  B. **Conversation-link partial commit no longer moves the source.**
     The outer-loop `consumed_prior_ids.add(prior_id)` only fires when
     a prior's outcomes are *all clean* (ROUTED / DUPLICATE_SKIPPED).
     A `[ROUTED, FAILED]` prior is now treated like a clash: flag the
     reply, leave both messages in the Inbox, do NOT add to
     `consumed_prior_ids`. Previously: the partial-commit prior got
     added to `consumed_prior_ids`, so when the outer loop later
     reached the prior message itself, it moved it to `processed_emails`
     and hid the failure.

  C. **Flagged-conversation-link priors are now quarantined across the
     outer loop.** Added `flagged_prior_ids: set[str]` alongside
     `consumed_prior_ids`. When `_process_prior_attachments` returns
     `flagged=True` (or the new partial-commit branch fires), the
     prior's id goes into this set. The outer-loop skip check now
     handles both: `consumed_prior_ids` -> skip+move; `flagged_prior_ids`
     -> skip+DON'T move. Previously: a flagged prior could later be
     picked up in the outer loop and re-processed via Branch A / C if
     its own subject or PDF text matched a plan, potentially routing
     its PDFs under a different plan than the reply pointed at.

  D. **`_flag_message_safely` returns bool and logs `FLAG_SET_FAILED:`.**
     The Outlook to-do flag is the operator-facing signal for this
     release; a silent Graph failure (permissions, rate-limit, schema)
     would degrade a dangerous clash into ordinary backlog. The wrapper
     now returns True/False and logs at error level with a grep-able
     prefix when the PATCH fails. Callers don't act on the return yet
     (the email-in-Inbox state is already correct from each branch's
     normal logic); the loud log is the escalation surface.

  E + F. **Workflow doc Edge-cases section updated** to match v0.10.0
     behaviour. Three drifted bullets fixed (multi-PDF emails, scanned
     PDFs, prior multi-PDF) plus new bullets for partial-commit and
     `FLAG_SET_FAILED:` semantics.

  G. **`plan_base` is robust to leading/trailing whitespace.** Now
     does `str(plan_norm).strip().upper()` before the regex. Current
     callers all pass already-normalised `PlanRow.plan_norm` so no
     production behaviour change, but the helper is public-shaped and
     was bait for a future caller. Added test cases for whitespace,
     digits-only, and prefix-only inputs.

  H. **Release notes (0.10.0) softened.** Replaced the overclaiming
     'previously routed correctly continue to route correctly' line
     with the actual trade: stricter matching can flag previously-
     routing edge cases, plus an explicit list of operator-visible

  I. **Test-input cleanup in `test_step1_decision.py`.** The
     suffix-variant test for `BCS2707A + BCS2707B` used a logically
     impossible classification (CLASH with the same plan as the

Tests: every existing test still passes. New whitespace cases added to

0.10.0  —  2026-05-12  —  PDF cross-validation + Outlook to-do flag in Step 1
--------------------------------------------------------------------------
Step 1 now treats every PDF as a second opinion on the subject's plan
match instead of trusting the subject blindly. Two real problems get
fixed: vendors who type the wrong plan in the subject while the PDF
has the correct one, and multi-PDF emails where each invoice belongs
to a different strata.

What's new:

  1. PDF text is scanned on every email where the subject (or body)
     matched a plan — not just on the no-match fallback. The scan
     reuses the in-memory bytes already downloaded for routing; no
     extra Graph calls.

  2. Each PDF is classified AGREE / EMPTY / AMBIGUOUS / CLASH against
     the subject's plan. `steps/step_1_intake.py::_decide_email_action`
     combines per-PDF classifications into one of:
       - ROUTE_AS_SUBJECT — every PDF agrees with the subject (or is
         a scanned image with no text). Current behaviour.
       - AUTO_SPLIT — multi-PDF email where every PDF identifies its
         own DISTINCT plan base (e.g. BCS 2707 + BCS 2800). Each PDF
         routes to its own manager; subject is ignored.
       - FLAG_AND_HOLD — strict-first default for everything else
         (single-PDF clash, consensus clash, suffix-variant clash,
         ambiguous text, empty-mixed-with-clash). Sets the Outlook
         "Flag as to-do" on the message (red flag in Outlook -> green
         checkmark when the operator marks complete), leaves the
         email in the Inbox, writes nothing to disk.

  3. Suffix-variant failsafe via `tools/_lib/plan_match.py::plan_base`:
     LMS4193C / LMS4193T / LMS4193S share base LMS4193, so a multi-PDF
     email split across these gets flagged for review (vendors confuse
     these easily). Same convention as `xls.base_plan_index`.

  4. Conversation-link recovery (reply-to-self) also classifies each
     prior PDF. The pre-0.10.0 limitation ("ALL PDFs in the prior get
     routed using the reply's plan, with only a count>1 warning") is
     gone. On clash/ambiguity, the reply is flagged and the prior-walk
     stops.

  5. `tools/_lib/graph.py::flag_message` — new helper. PATCH the
     message's `flag` property to `{flagStatus: flagged}`. The
     operator clears it in Outlook (which sets `flagStatus =
     complete`) when handled.

What's the same:

  - All-or-nothing rule for unmatched emails (subject + body + PDF
    all miss): unchanged.
  - Receive-stamping, duplicate ledger, conversation-link walk
    direction (newest-first): unchanged.
  - Non-PDF attachments still park in `_Unmatched/Invoices/` on route
    paths; on FLAG_AND_HOLD they stay in the email (strict-first).
  - No new env vars, no new step files, no folder layout changes.

Backwards-compatibility: file paths, env vars, step contracts
unchanged. The matching path is **stricter, not stricter-and-different**:
most emails that previously routed correctly still route correctly.
Emails that previously routed but had ambiguous PDF text, suffix-variant
attachments, or marginal cross-checks may now flag for review instead —
that's the intended trade. Admins deploying this version should expect
a small bump in flagged-email volume during the first weeks while the
strata managers' labelling stabilises.

Known operator-visible scenarios (see also 0.10.1 fixes below):
  - Batch partial commits (PDF #1 routed, PDF #2 fails) leave the email
    in the Inbox with the red flag set (since 0.10.1) and a
    `partial commit on '<subject>'` error log line.
  - Graph flag-set failures (rare; permissions / rate-limit / schema)
    log `FLAG_SET_FAILED:` at error level and leave the email in the
    Inbox WITHOUT a red flag. Admins should grep for this prefix in
    the daily log if the flagged-email queue feels under-populated.

Strict-first stance: flag now, relax later if strata managers prove
their labelling is reliable. Specific paths for relaxation are noted
in `To-Speak-About.txt`'s 2026-05-12 top entry.

Settles `To-Speak-About.txt` duplicate-detection Q3 ("Should the

  external API.

entry points (Steps 3/5/6: manual drops, retries) pass a blank

Vendors sometimes send the same invoice 2-3 times in a week. The pipeline
now catches that before it wastes manager or accountant attention. A
persistent ledger at `_state/invoice_fingerprints.csv` records every
unique invoice that flows through; same-day, next-day, and weeks-later
duplicates are all detected.

Two-layer fingerprint per PDF:
  Layer A  SHA-256 of the PDF bytes. Catches verbatim resends ("Send"
           pressed twice; accounting system auto-resent). Free,
           deterministic, no false positives.
  Layer B  (plan_norm, invoice_number, amount_cents) extracted from
           the PDF text. Catches "vendor regenerated the same invoice
           with different bytes". Invoice number is recognized via
           labels like `Invoice #`, `Invoice No.`, `INV:`. Amount via
           `Total Due` / `Amount Due` / `Balance Due` / `Grand Total`.

Vendor name is NOT extracted (avoids per-invoice LLM cost; invoice
numbers don't collide across vendors in practice).

Hook points (where the check fires):
  - Step 1 intake — inside `_route_pdf`. Primary catch (~95% of dups).
  - Step 3 pdf_sort — after `read_bytes()`. Catches duplicates from ZIP
    extraction or operator manual drops into `_Unmatched/`.
  - Step 5 to_ap — after `read_bytes()` in `_transfer_phase`. Safety
    net for PDFs the manager dragged straight into `Approved/`.
  - Step 6 paid_archive — after archive write. Enriches the ledger
    row with `archive_path` for the daily summary.

Email routing (Step 1):
  - All attachments duplicate => `Inbox/duplicate_emails`
  - Mix of fresh routes + duplicates => `Inbox/processed_emails`
  - All fresh routes => `Inbox/processed_emails` (unchanged)
  - Any genuine failure => stay in Inbox (unchanged all-or-nothing)

  Operator setup: create `Inbox/duplicate_emails` subfolder in OWA once,
  same way `Inbox/processed_emails` was created.

PDF routing on duplicate hit:
  - Step 1: nothing written to disk. The email itself preserves the
    bytes in `Inbox/duplicate_emails`; the original archived PDF is
    referenced via `archive_path` in the ledger. Zero new local files.
  - Step 3: source renamed `Processed-<ts>-DUPLICATE-<name>.pdf` so
    Step 3's glob excludes it next run.
  - Step 5: source renamed `Processed - DUPLICATE - <name>.pdf` in the
    manager's `Approved/` folder.

Daily duplicate-summary email:
  Step 6 at 07:00 emits "Duplicate invoice detections: N — <date>" to
  `notify_email()` when at least one fingerprint had its `dup_count`
  incremented today. Zero-duplicate days produce no email. Each entry
  lists plan, original archive path or current pipeline location,
  extracted invoice number, amount, first-seen date, and total times
  seen.

Ledger:
  Single CSV at `_state/invoice_fingerprints.csv`, one row per unique
  fingerprint, forever retention. Upsert-in-place (not append-only) so
  state transitions overwrite the row and only `dup_count` accumulates.

  Schema (10 columns):
    first_seen_date, sha256, plan_norm, invoice_number, amount_cents,
    archive_path, current_stage, last_seen_date, dup_count, last_dup_date

  Stages: intake, manager_queue, ap_queue, archived, quarantined,
  overridden, superseded. `superseded` marks an overridden row whose
  Layer B semantic key was matched by a regenerated PDF — the old sha
  is retained as audit history and blocks future arrivals of those
  exact bytes; the new sha gets a fresh row.

  Sizing budget at 45-60 invoices/day:
    Year 1:    ~2 MB  (15K rows)
    Year 5:   ~11 MB  (75K rows)
    Year 10:  ~22 MB  (150K rows)
  Excel-friendly indefinitely. No archive rotation needed.

  Audit trail (subject lines, message IDs, per-decision logs) stays in
  the existing per-step daily logs. The fingerprint file is purely the
  dedup key + pointer to the original.

  Corrupted-ledger handling: `load()` raises ValueError on a malformed
  row; the step halts the day. Silent-treat-as-empty would re-enter
  every previously-archived invoice into the pipeline.

Concurrency model — transactional read-modify-write:
  Every mutator (`upsert`, `increment_dup_count`, `update_stage`,
  `consume_override_and_insert`) is transactional. The lock spans the
  full read-modify-write cycle, NOT just the final write:

    1. Acquire portalocker.Lock
    2. Re-read disk into a fresh in-memory snapshot
    3. Apply the mutation to that fresh state
    4. Write the whole CSV via safe_io.atomic_write_bytes
    5. Release the lock
    6. Commit the new state to the in-memory caches

  In-memory state is committed ONLY after the lock is released and the
  write succeeded. If the write raises, the in-memory caches stay at
  the pre-mutation snapshot so the caller can retry with a coherent
  view (no phantom rows). If two processes mutate the same ledger
  (hand-run + cron overlap), the second transaction sees the first's
  writes during its locked re-read — no lost-update race.

  Reads (`find_by_hash`, `find_by_semantic_key`) consult the in-memory
  cache and may be stale by up to one mutation cycle; the subsequent
  mutation re-reads under the lock so a stale "not seen" doesn't cause
  data loss in the ledger itself (see "Known limitation" below for the
  narrow PDF-write race).

  `find_by_semantic_key` excludes rows in `overridden` and `superseded`
  stages so those rows don't block new arrivals via Layer B. The
  explicit `find_overridden_by_*` helpers surface override rows for
  the consumption path.

`_route_pdf` signature change:
  Returns `RouteOutcome` (`routed` / `duplicate_skipped` / `failed`)
  instead of a bare bool. All three Step 1 callers
  (`_process_self_attachments`, `_process_prior_attachments`,
  `_process_pdf_text_fallback`) updated to branch on the enum. The
  Pass 1 / Pass 2 architecture in `_process_pdf_text_fallback` extended:
  Pass 1 records a non-mutating duplicate check per PDF; Pass 2's
  `can_route_all` gate accepts "matched and duplicate" as auto-
  handleable (the system knows what to do — skip it). Pass 2 commits
  to ledger only when the email is fully auto-handleable, so a
  short-circuited email does NOT pollute `dup_count`.

Override consume semantics (one-shot):
  An override is consumed differently depending on which layer hits:

  Layer A (same bytes re-arrive):
    `_check_dup_status` sees `current_stage=overridden` and treats it
    as "route normally". The post-route `upsert` overwrites the
    override row in place with `current_stage=manager_queue`. The
    override is now spent; future arrivals of these exact bytes
    block as normal duplicates.

  Layer B (regenerated bytes, same semantic key):
    `_check_dup_status` surfaces the override row separately via
    `find_overridden_*`. After a successful route, the caller invokes
    `ledger.consume_override_and_insert(old_sha, new_row)` which
    atomically: (a) verifies the old row is still `overridden`,
    (b) marks it `superseded`, (c) inserts the new sha row. The old
    sha is retained as audit history; a future re-arrival of the OLD
    bytes hits a `superseded` row and IS blocked as a duplicate.

  Double-consume protection: `consume_override_and_insert` re-reads
  the old row's stage inside the locked transaction. If it's no
  longer `overridden` (already consumed by another process or call),
  it raises ValueError. Callers catch this and fall back to a plain
  `upsert` so a race between two regen-arrivals doesn't blow up.

Operator CLIs (on-demand, not scheduled):
  python tools/dup_override.py <sha256_prefix> --reason "..."
    Marks a fingerprint as `overridden` so the next arrival routes
    through normally (one-shot — consumed by the next pipeline pass).
    Reason is required and logged to `logs/dup_override_<date>.log`.
    Prefix must be hex-only and >= 12 characters (or a full 64-char
    SHA) to avoid typo-induced wrong-row matches; validation runs
    before ledger load so the error is clear even without env config.

  python tools/dup_reconcile.py [--tsv out.tsv]
    Walks every pipeline folder, hashes every PDF, cross-checks
    against the ledger. Reports orphans (on disk, not in ledger),
    stale archived rows (ledger -> missing file), multi-arrival
    fingerprints (`dup_count > 0`), and orphan `.tmp.<pid>` files
    left behind by interrupted atomic writes. Read-only; does NOT
    mutate.

Edge cases handled:
  - Same-day double arrival in one Step 1 run (in-memory ledger reflects
    the prior insert before the second lookup).
  - Duplicate weeks after archival (summary email references the
    archive path verbatim).
  - Conversation-link "you already sent this" (reply + prior both
    move to `duplicate_emails`).
  - Manager manual drop into `Approved/` (Step 5 catches via safety
    net; orphan rows are self-healing — new row inserted at ap_queue).
  - OLD bytes re-arrive after Layer B override consume: blocked as
    duplicate (superseded row remains active for Layer A matching).
  - Two concurrent processes both consuming the same override: only
    the first succeeds; the second falls back to plain upsert.
  - Cross-process mutation race (hand-run + cron): transactional RMW
    ensures no lost ledger updates.
  - Ledger missing (first run): empty ledger, auto-created.
  - Ledger corrupted: halt with clear error.
  - Atomic-rewrite mid-write crash: previous good copy survives;
    in-memory state untouched so caller can retry coherently.

Known limitation:
  Cross-process check-then-write TOCTOU. Steps consult the ledger
  via in-memory lookups BEFORE writing a PDF to disk. The mutation
  itself is transactional under the lock, but the preceding check
  is not. In the narrow window between "Process A read the ledger
  and concluded the fingerprint was new" and "Process A's upsert
  acquired the lock", Process B can insert a row for the same
  fingerprint. Both A and B then write their PDF to disk. Realistic
  exposure under normal Task Scheduler timing (06:00 / 06:20 /
  06:40 / 07:00) is essentially zero; the race requires a hand-run
  overlapping a cron AND both runs picking up the same fingerprint.
  Documented in `workflows/duplicate_detection.md` under "Known
  limitation"; a complete fix would require restructuring the PDF
  write path to reserve the ledger entry before writing.

0.8.0  —  2026-05-11  —  Step 1: Inbox-first recovery for unidentified invoices
--------------------------------------------------------------------------
Step 1 now identifies the Strata Plan from email subject, email body
(bodyPreview), or PDF text inline — instead of leaving subject-only-
unmatched emails for Step 3 to retry 20 minutes later. The downstream
effect: unidentified emails STAY IN THE INBOX (not moved to
`processed_emails`), so the operator's reply-to-self recovery workflow
works the way it did in the N8n flow.

Recovery workflow (documented in README "Recovery workflow" section and
docs/client_training_brief.md):
  1. Operator sees an unmatched email sitting in the Inbox.
  2. Hits Reply. Changes To: from vendor's address to
     `testinvsml@stratacomgmt.com`. Edits subject to include the strata
     number. Sends. **No PDF re-attachment needed.**
  3. Next morning Step 1 sees the reply, matches the subject, looks up
     `conversationId` via Microsoft Graph, pulls the PDF from the
     prior message in the thread, routes it, moves both messages to
     `processed_emails`.

Matching order (first hit wins, per Inbox message):
  1. Subject  — `pick_from_subject(message.subject)`
  2. Body     — `pick_from_subject(message.bodyPreview)` (first ~255
                chars; full-body matching is a future enhancement)
  3. PDF text — `match_from_pdf_text(extract_full_text(blob))` against
                the in-memory PDF bytes
  4. Conversation-link (only when subject/body matched + no attachments)
     — walk `conversationId` newest-first, route from the first prior
     message that has a routable PDF

All-or-nothing rule for emails without a subject/body match:
  - Every PDF must text-match a plan AND there must be no non-PDF/ZIP
    attachments — only then do we route and move the email.
  - Otherwise the entire email stays in the Inbox with nothing written
    to disk. Honors "Inbox is the single source of truth" — we never
    silently leave anything in `_Unmatched/` that the operator wouldn't
    notice.

Codex adversarial review caught and we fixed:
  - Mixed matched/unmatched PDFs no longer cause silent data loss (the
    initial fix saved unmatched PDFs to `_Unmatched/` but operators
    don't check there, so we switched to true all-or-nothing).
  - Unmatched non-PDF/ZIP-only emails no longer duplicate-save every
    run (they stay in Inbox now, no disk write).
  - Conversation-link now walks ALL priors newest-first looking for a
    routable PDF, not just the most recent attachment-bearing one (the
    most recent might have only a Word doc).
  - Multiple corrected replies in the same run can't re-route the same
    prior twice — `consumed_prior_ids` is tracked across the run and
    excluded from the prior-finder.
  - Conversation lookup failures are logged with conv_id context, not
    silently swallowed.
  - `_route_pdf` early-returns False (logged) if `plan_row.manager_name`
    is empty, instead of raising and aborting the whole step.
  - Multi-PDF prior messages log an error-level warning that all PDFs
    are being routed to the reply's plan (so operator can spot a
    misroute in the daily log).

Known limitations (logged in To-Speak-About.txt for client direction):
  - Multi-strata emails (one email containing PDFs for different
    properties without any plan in subject/body): operator's reply-to-
    self routes all PDFs to one strata; operator must manually
    re-route misrouted invoices from the wrong manager's folder.
  - ZIPs in unidentified emails: stay in Inbox, no auto-extraction.
    Operator handles manually. Trade-off accepted to avoid duplicate-
    save loop.
  - bodyPreview is the first ~255 chars of the email body, not the
    full body. Future enhancement: fetch full body + strip HTML.

0.7.0  —  2026-05-11  —  Step 7: Monthly Invoice Aggregator
--------------------------------------------------------------------------
New monthly step. For each active strata plan, merges every Step-6-archived
invoice from a target month into a single combined PDF and moves the source
PDFs into a dated subfolder. End state per plan, per month: one clean
`Summary - <MM> - <plan> <Month> <YYYY> inv.pdf` in the plan folder, plus
an audit trail in `Strata_Plans/<plan>/Processed/<YYYY-MM>/`.

Schedule:
  Operator-chosen day in Task Scheduler (typically 5th–10th of the new
  month). The script always aggregates the PREVIOUS calendar month by
  default (America/Vancouver). `--month YYYY-MM` overrides; future months
  and >24-mo-historical values are rejected before the per-step lock is
  acquired.

CLI:
  python steps/step_7_aggregate.py                  # previous month, all plans
  python steps/step_7_aggregate.py --month 2026-04  # explicit month
  python steps/step_7_aggregate.py --plan BCS1234   # single plan
  python steps/step_7_aggregate.py --dry-run        # write nothing
  python steps/step_7_aggregate.py --force          # bypass ledger short-circuit

Audit ledger (`_state/monthly_aggregations.csv`):
  Single append-only CSV that records every (plan, target-month) attempt.
  Step 7 consults the ledger before doing work and surfaces the count on
  startup ("Ledger: 0 of 47 active plans aggregated for April 2026 — will
  process all"). The ledger answers "did April actually finish for every
  plan?" from one file and protects against accidental re-runs.

  Statuses: `aggregated`, `aggregated_late`, `dry_run`,
  `skipped_already_done`, `skipped_no_files`, `skipped_no_folder`, `error`.
  `dry_run` is informational and explicitly does NOT count as done — a
  follow-up real run still produces `aggregated`, not `aggregated_late`.

Late-check semantics:
  If Step 7 has already aggregated a month and a new invoice for that
  month appears later (Step 6 archived it after Step 7 ran), the next
  Step 7 run writes an additive `Summary - ... inv (1).pdf` rather than
  overwriting the original. Ledger row: `aggregated_late`. Nothing is
  silently dropped, nothing is silently overwritten. Whether the
  destructive re-merge would be preferable is logged as a policy
  question in `To-Speak-About.txt`.

Idempotency:
  Five rules consulted per plan after a candidate scan — see
  `workflows/step_7_aggregate.md`. The two load-bearing ones:
    1. Ledger says done + Summary file present + Processed/{YYYY-MM}/
       populated → `skipped_already_done`.
    2. Ledger says done + Summary OR Processed/ missing
       → `error` (`ledger-filesystem disagreement`), refuse to process
       until operator runs with `--force` after triage.

  Rule 1 verifies the SUMMARY FILE is actually present, not just that
  Processed/ has files — a deleted/corrupt Summary used to be silently
  treated as done.

Move-failure rollback:
  If any source PDF fails to move into Processed/ (e.g. Acrobat has it
  open with an exclusive lock), Step 7 rolls back the successful moves
  AND deletes the just-written Summary. The plan goes back to a clean
  "not done" state, ledger row is `error` with the failure details, and
  the operator can investigate without producing a `Summary ... (1).pdf`
  with duplicate pages on the next run.

--force from Processed/:
  When the plan-folder root is empty (operator deleted the Summary to
  redo from scratch) AND `--force` is set, Step 7 falls back to scanning
  `Processed/{YYYY-MM}/` for matching files and regenerates the Summary
  from those. No move is performed since the files are already where
  they belong. Ledger status: `aggregated_late` with notes mentioning
  the source. Without this fix, `delete summary + run --force` produced
  `skipped_no_files` and did nothing.

Snapshot self-refresh:
  Step 7 calls `refresh_snapshot()` at startup before
  `require_fresh_snapshot()`. The monthly trigger may fire on days
  Step 1's Mon–Fri schedule did not run (weekend, holiday). Refresh
  failure is non-fatal as long as today's marker is present (no
  regression on business-day runs).

`--force` classification:
  `--force` no longer suppresses the `aggregated_late` classification.
  Once a (plan, month) has been aggregated, every subsequent run for
  that pair is `aggregated_late`, regardless of whether `--force` was
  needed to get there. Keeps the audit trail unambiguous: the FIRST
  aggregation is `aggregated`; everything after is `aggregated_late`.

Sort-key bucketing:
  `_check_sort_key` now buckets alpha-prefixed checks ("AB-123",
  "WIRE-9") separately from pure-numeric checks ("12345"). Pure-numeric
  sort first by integer value; alpha-prefixed checks sort by prefix,
  then by their internal number. Previously "AB-123" would sort at
  integer position 123 among numeric checks — fine for purely-numeric
  plans, wrong as soon as a strata mixes regular checks with wires.

Parser hardening:
  `parse_archive_filename()` defensively rejects names starting with
  `Summary -`, `Processed -`, or `Paid -` so a permissive call site
  can never accidentally re-aggregate Step 7's own output.

Ledger zero-byte handling:
  Ledger `append()` writes the CSV header when the file is missing OR
  exists but is zero-bytes (Excel crash mid-save). Previously the first
  data row would silently become the de-facto header.

Dry-run email:
  Dry-run sends a distinct "DRY RUN — Monthly aggregation" email subject
  that lists which plans WOULD have been aggregated, instead of the
  ambiguous "Monthly aggregation: 0 plans" message of a real-but-empty run.

0.6.0  —  2026-05-11  —  Strataplan_List.xlsx working-copy snapshot
--------------------------------------------------------------------------
Step 1 now refreshes `_state/strataplan_list_snapshot.xlsx` at 06:00 and all
plan-consuming steps (1, 3, 4, 5, 6) read that snapshot instead of the master.
The bundler reads the snapshot too, so diagnostic output matches what the day
actually saw.

The realistic baseline at 06:00 is that the operator left the master open in
Excel from the previous day's editing session. The snapshot module reads the
master via `CreateFileW` with `FILE_SHARE_READ | FILE_SHARE_WRITE |
FILE_SHARE_DELETE` (via stdlib `ctypes`, no new dependency). Excel itself
opens xlsx files with those share flags, so Windows allows a matching-share
read through even while Excel holds the file. The bytes we read are whatever
Excel last saved to disk; unsaved in-memory edits are invisible, which matches
the "edit, save, close" protocol.

Halt-on-failure: if Step 1's refresh fails for any reason (master missing,
read error, bytes don't validate as XLSX), Step 1 returns 1 and the marker
sidecar stays at yesterday (or is never written). Steps 3, 4, 5, 6 read the
marker; if it's not today's date they refuse to run and surface
`snapshot is not today's — refusing to run` in `daily_summary.csv`. Stale
routing is unsafe — a re-routing on the spreadsheet (manager vacation, new
AP) must take effect the same day, not the next. Step 2 (unzip) is
independent of the snapshot and still runs.

Verify-before-publish atomicity (post-review hardening):
  The first iteration validated the snapshot AFTER atomic_write_bytes had

  Regression test in tests/test_strataplan_snapshot.py covers this exactly.

0.5.0  —  2026-05-11  —  Step 6 archive filename: client's date-bearing convention
--------------------------------------------------------------------------
Step 6 now writes archive filenames in the format the client asked for:

    {check} - {MM} - {PLAN} {MonthName} {YYYY} inv.pdf
    e.g.   12345 - 03 - BCS1234 March 2026 inv.pdf

The original vendor filename is dropped from the archive name — every
component is derived from values the accountant already enters on the
Paid stamp (Check Number + Date) and values we already know
(plan_norm, the normalised no-space form). Earlier steps and the
`Processed - <original>.pdf` marker in the AP folder are unchanged —
operators still recognise files at intermediate stages by the name
they saw in their inbox.

Date extraction is back. tools/_lib/stamp_read.py was stripped of date
extraction in 0.1.4 because it grabbed vendor "Date:" labels in the
invoice body and returned junk. Re-introducing it is safe now that
0.3.0 narrowed the stamp search to the PAID region (`_locate_paid_region`)
— the same protection that fixed Check Number extraction now protects
the Date extraction. PaidStampValues gained `paid_date` and
`has_paid_date` alongside `check_number`; the regex fallback uses the
same ±300-char PAID-proximity guard.

New helper `parse_paid_date(raw) -> (month, year) | None` accepts every
format an accountant is likely to type:

  * `MAY 08 2026` / `May 08 2026` — the default `render_paid_stamp`
    output.
  * `May 8, 2026` / `March 31 2026` — long-form with optional comma.
  * `2026-05-08` — ISO; unambiguous.
  * `MM/DD/YYYY` / `DD/MM/YYYY` — if one position exceeds 12, that's
    the day; otherwise MM/DD/YYYY default.
  * `MM/DD/YY` — REJECTED. 2-digit years are not guessed; the file
    flags as unmatched instead.

A new unmatched reason — "Could not read Date from Paid stamp" — joins
the existing reasons in the Step 6 unmatched summary email. Same
operator remediation as the Check Number row: open in Acrobat, confirm
both fields show a value AND that the file is flattened.

Critical: `_archive_one` and `_archive_exists_for` in
steps/step_6_paid_archive.py now build the archive name through a
single shared helper `_build_archive_name(check, month, year,
plan_norm)`. If the two ever diverge, every previously-processed file
in the AP folder would throw a false "stale processed marker" error,
silently surfacing as unmatched. The helper keeps them locked.

Existing archives sitting in `Strata_Plans/<plan>/` with the old
`{check} - {original}` naming are untouched — the new convention only
applies to files going forward. Renaming history would be a destructive
irreversible operation and isn't part of this ask.

Tests (tests/test_stamp_read.py extended; all 15 cases green):

  - 11 unit cases for `parse_paid_date` covering every accepted format,
    the day-first auto-detect (13/08/2026 → August), and the rejection
    paths (empty, garbage text, 2-digit years).
  - The three real mock invoices in reference/stamp_samples/ now round-
    trip BOTH check number and date through `render_paid_stamp` →
    manual-flatten (drawing the date value at row 0, check at row 1) →
    `extract_paid_stamp_values` → `parse_paid_date`.
  - The 0.3.0 decoy synthetic case was extended with a parallel
    "Date: JAN 01 1999" label in the invoice header. The extractor
    correctly returns the in-stamp `MAY 08 2026` and `55555`, proving
    region narrowing protects the date the same way it protects the
    check number.

- New PAGE_MARGIN_PT constant (36pt = 0.5 inch) in tools/_lib/stamp.py.

New behaviour vs. the N8n flows:
  - Step 5: drops the "Approved - " prefix when copying to the AP folder
    and applies an editable Paid stamp (blue, two AcroForm text fields).
  - Step 6: processes every PDF in Paid_Invoices (no "Paid -" filter),
    reads the Check Number from the flat PDF, and prefixes the destination
    filename with it.

Stamp module:
  - Re-implemented from scratch (no dependency on the 192.168 service).
  - Whitespace placement via numpy integral-image search, lower-right bias.
  - Received stamp: red, 7 rows, 2 fixed (date, plan) + 5 editable AcroForm
    fields (gl_code, chargeback, chargeback_to, amount, approved).
  - Paid stamp: blue, 3 rows, header "Paid" + 2 editable fields (date,
    check_number).

Concurrency safety:
  - Atomic writes (.tmp.<pid> + os.replace) for every PDF/XLS the steps
    produce.

  - Schedule stagger preserved (06:00 / 06:10 / 06:20 / 06:30 / 06:40 /
